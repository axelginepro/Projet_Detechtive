Certification AIS (Niveau 6)

1. Contexte & Objectifs

Scénario : Pilotage d'un projet d'infrastructure global : conception de l'architecture réseau sécurisée et intégration complète de la stack technique (Web, Data, Fichiers) dans une approche Secure by Design.

  • Confidentialité : Chiffrement intégral des flux (HTTPS/SSL), segmentation stricte par VLANs et sécurisation du stockage par cloisonnement.
  • Disponibilité : Architecture robuste — grâce au cloisonnement VLAN, la compromission d'un élément ne propage pas l'attaque au reste du réseau.
  • Traçabilité : Centralisation des logs (Wazuh SIEM) depuis un VLAN Management dédié et authentification unique (Active Directory).

2. Architecture Infrastructure

Segmentation réseau — 5 zones de sécurité

Le switch IOU (L3) assure le routage inter-VLAN entre les 5 zones internes (ip routing, SVI par VLAN) et porte les passerelles internes. pfSense est positionné en coupure entre Internet et le switch : il assure le filtrage périmétrique, le NAT et le VPN. Ils sont interconnectés via un réseau de transit point-à-point dédié (10.0.0.0/30) pour isoler le routage interne : pfSense (10.0.0.1) ↔ Switch L3 (10.0.0.2). pfSense utilise des routes statiques pour joindre les VLANs via le switch.

Pourquoi cette segmentation ?

Serveur Web en DMZ

Le Web Server (192.168.10.10) est le seul élément exposé à Internet via NAT pfSense. En l'isolant dans VLAN 10, une compromission n'expose ni l'AD, ni la BDD, ni les postes clients.

AD isolé (VLAN 40)

L'Active Directory (192.168.10.34) est le composant le plus critique. Un attaquant qui compromet la DMZ ne peut pas atteindre l'AD — tout flux VLAN 10 → VLAN 40 est bloqué par pfSense.

Wazuh isolé (VLAN 50)

Le SIEM Wazuh (192.168.10.42) est en VLAN dédié. Même si la DMZ est compromise, l'attaquant ne peut pas désactiver la supervision — aucun flux entrant VLAN 10 → VLAN 50 n'est autorisé.

3. Gestion des Identités (Active Directory)

Moindre Privilège (Least Privilege)

Le principe fondamental de sécurité appliqué au projet.

  • Structure AD : Organisation par Unités d'Organisation (OU) pour séparer les Agents des Administrateurs.
  • Droits Restreints : Les utilisateurs standard n'ont aucun droit d'administration sur les serveurs ou les postes de travail.
  • Comptes de Service : L'application Web utilise un compte AD dédié avec des droits minimaux, uniquement capable d'écrire dans les dossiers de dépôt.
  • PKI Interne (AD CS) : Le serveur AD gère l'Autorité de Certification pour chiffrer les flux internes (Web/MariaDB) et distribuer la confiance via GPO.

Serveur de Fichiers Sécurisé

Isolation stricte des données sensibles.

  • Dossiers Dédiés : Chaque agent dispose d'un dossier partagé personnel sur le File Server (192.168.10.20, VLAN 20).
  • Permissions NTFS : ACLs configurées pour que l'Agent A ne puisse ni lire ni voir le dossier de l'Agent B.
  • Mappage Dynamique : Au login, seul le lecteur réseau correspondant à l'utilisateur connecté est monté.

4. Développement & Backend (Détail)

Apache (XAMPP/Windows) PHP 8.x MariaDB Windows Server 2022 SMB

Serveur Web — VLAN 10

IP fixe : 192.168.10.10. Seul point exposé à Internet.

  • Rôle Pivot : Orchestre les demandes entre l'utilisateur, la BDD (VLAN 20 — port 3306) et le File Server (VLAN 20 — SMB).
  • Sessions : Gestion stricte via session_start(). Redirection si non authentifié.
  • Sécurité : Validation des entrées et whitelist des fichiers uploadés.

Base de Données — VLAN 20

IP fixe : 192.168.10.21. Inaccessible depuis Internet.

  • Connexion SSL : Le client PHP utilise PDO::MYSQL_ATTR_SSL_CA pour valider l'identité du serveur MariaDB.
  • Isolation : VM dédiée en VLAN 20 — aucun accès direct depuis VLAN 10 hors port 3306.
  • Failover : try/catch PHP pour basculer en mode dégradé si SSL échoue.

File Server — VLAN 20

IP fixe : 192.168.10.20. Stockage des preuves.

  • Protocole : Partage Windows (SMB) monté dynamiquement depuis VLAN 10.
  • Bridge PHP : Commande net use pour monter le lecteur réseau à la demande.
  • Permissions : Compte de service AD dédié (Least Privilege) pour l'écriture uniquement.

5. Sécurité Applicative (AppSec)

Protection du Code

  • Anti-SQLi : Usage exclusif de PDO et des requêtes préparées.
  • Anti-RCE : Renommage forcé des fichiers uploadés (preg_replace) pour éviter l'exécution de scripts.
  • Feedback SSL : Indicateur visuel "Cadenas Vert/Rouge" informant l'agent de l'état du chiffrement (Ssl_cipher).

Sécurité Infrastructure

  • Switch IOU (L3) : Routage inter-VLAN via SVI dédiées — ip routing activé, route par défaut vers pfSense (10.0.0.1).
  • Pare-feu pfSense : Filtrage périmétrique stateful, NAT (Web → 192.168.10.10), VPN WireGuard UDP 51820. Règles whitelist strictes, aucune règle "any to any".
  • SIEM Wazuh (192.168.10.42) : Remontée d'alertes en temps réel depuis VLAN 50 dédié, inaccessible depuis la DMZ.
  • Traçabilité (Patch Management) : Les accès Internet pour les mises à jour sont autorisés, mais intégralement loggués vers le SIEM pour prévenir toute exfiltration.
  • Hardening : Désactivation des services inutiles et isolation physique des ports (VLAN 999 en shutdown).
    ▶ Perspective d'évolution : Déploiement futur du protocole HSRP/VRRP pour redonder le switch cœur et éliminer le point de défaillance unique (SPOF).

6. VPN WireGuard (Accès Admin Distant)

Paramètres

  • Protocole : UDP, Port 51820 sur l'interface WAN de pfSense.
  • Plage VPN : 10.10.10.0/24 (réseau virtuel WireGuard).
  • Accès autorisé : VLAN 40 (Active Directory) et VLAN 50 (Management & Sécurité).
  • Accès interdit : VLAN 30 (postes clients) — principe du moindre privilège.

Justification

Le VPN est réservé aux administrateurs IT en télétravail — pas aux agents. Il permet de gérer pfSense, l'AD et Wazuh sans ouvrir RDP/SSH directement depuis Internet. Les agents accèdent au réseau physiquement depuis leurs postes Windows en VLAN 30.
