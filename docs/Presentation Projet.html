1. Contexte & Objectifs

Scénario : Pilotage d'un projet d'infrastructure global : conception de l'architecture réseau sécurisée et intégration complète de la stack technique (Web, Data, Fichiers) dans une approche Secure by Design.

  • Confidentialité : Chiffrement intégral des flux (HTTPS/SSL), segmentation stricte par VLANs et sécurisation du stockage par cloisonnement.
  • Disponibilité : Architecture robuste — grâce au cloisonnement VLAN, la compromission d'un élément ne propage pas l'attaque au reste du réseau. Sauvegardes isolées en VLAN 60 dédié.
  • Traçabilité : Centralisation des logs (Wazuh SIEM) depuis un VLAN Management dédié et authentification unique (Active Directory).

2. Architecture Infrastructure

Segmentation réseau — 6 zones de sécurité

▶ Architecture : Router-on-a-Stick (802.1Q)

Le switch IOU opère en L2 pur (commutation uniquement, pas de routage inter-VLAN). pfSense centralise l'ensemble du routage, du filtrage et du NAT via 6 sous-interfaces 802.1Q sur son interface LAN physique. Un trunk 802.1Q unique relie le switch à pfSense — pfSense porte toutes les passerelles VLAN.

Topologie GNS3

  • Switch IOU (L2) : Trunk 802.1Q unique vers pfSense — pas de routage interne.
  • VLAN 10 (DMZ) : Serveur Web exposé, isolé du réseau interne.
  • VLAN 20 : File Server & BDD, jamais joignables depuis Internet.
  • VLAN 40 : AD isolé — inaccessible depuis la DMZ par règle pfSense.
  • VLAN 50 : Wazuh collecte les logs de tous les VLANs.
  • VLAN 60 : NAS Backup isolé — reçoit uniquement, n'initie aucune connexion.

Règles de Filtrage pfSense

WAN → VLAN 10 : NAT ports 80/443 → .10

WAN → pfSense : ALLOW UDP 51820 (WireGuard)

VLAN 10 → VLAN 20 : ALLOW 3306 (DB) & 445 (SMB)

VLAN 10 → VLAN 40/50/60 : DENY ALL

VLAN 30 → VLAN 40 : ALLOW 389/636 (LDAP/S)

VLAN 30 → VLAN 20 : ALLOW 445 (SMB)

VLAN 20/40 → VLAN 60 : ALLOW backup (unidirectionnel)

VLAN 60 → Tous VLANs : DENY ALL (passif)

VLAN 60 → VLAN 50 : ALLOW 1514/1515 (logs Wazuh)

LAN internes → Internet : ALLOW sortant (loggué SIEM)

3. Gestion des Identités (Active Directory)

Moindre Privilège (Least Privilege)

Le principe fondamental de sécurité appliqué au projet.

  • Structure AD : Organisation par OU pour séparer les Agents des Administrateurs.
  • Droits Restreints : Les utilisateurs standard n'ont aucun droit d'administration sur les serveurs ou postes.
  • Comptes de Service : L'application Web utilise un compte AD dédié avec des droits minimaux.
  • PKI Interne (AD CS) : Le serveur AD gère l'Autorité de Certification pour chiffrer les flux internes.

Serveur de Fichiers Sécurisé

Isolation stricte des données sensibles.

  • Dossiers Dédiés : Chaque agent dispose d'un dossier partagé personnel sur le File Server (VLAN 20).
  • Permissions NTFS : ACLs configurées pour l'isolation complète entre agents.
  • Mappage Dynamique : Au login, seul le lecteur réseau correspondant est monté via GPO.

4. Développement & Backend (Détail)

Apache (XAMPP/Windows) PHP 8.x MariaDB Windows Server 2022 SMB / 445 PDO + SSL

Serveur Web — VLAN 10

IP fixe : 192.168.10.10. Seul point exposé à Internet via NAT.

  • Rôle Pivot : Orchestre les demandes vers la BDD (3306) et le File Server (445) en VLAN 20.
  • Sessions : Gestion stricte via session_start() et redirection si non authentifié.
  • Sécurité : Validation des entrées et whitelist des fichiers uploadés.

Base de Données — VLAN 20

IP fixe : 192.168.10.21. Inaccessible depuis Internet.

  • Connexion SSL : Le client PHP utilise PDO::MYSQL_ATTR_SSL_CA pour valider l'identité MariaDB.
  • Isolation : VM dédiée en VLAN 20 — aucun accès direct depuis DMZ hors port 3306.
  • Failover : try/catch PHP pour basculer en mode dégradé si SSL échoue.

File Server — VLAN 20

IP fixe : 192.168.10.20. Stockage des preuves en zone protégée.

  • Protocole : Partage Windows (SMB port 445) monté dynamiquement depuis VLAN 10.
  • Bridge PHP : Commande net use pour monter le lecteur réseau à la demande.
  • Permissions : Compte de service AD dédié — écriture sur dossier restreint.

5. Sécurité Applicative (AppSec) & Infrastructure

Protection du Code

  • Anti-SQLi : Usage exclusif de PDO et des requêtes préparées.
  • Anti-RCE : Renommage forcé des fichiers uploadés pour bloquer les scripts malveillants.
  • Feedback SSL : Indicateur visuel informant l'agent de l'état du chiffrement.

Sécurité Infrastructure

  • Switch IOU (L2) : Commutation pure — Ports inutilisés en VLAN 999 + shutdown.
  • Pare-feu pfSense : Routage inter-VLAN centralisé, filtrage stateful et VPN WireGuard.
  • SIEM Wazuh : Alertes temps réel depuis VLAN 50 dédié — logs unidirectionnels.
  • Hardening : Désactivation des services inutiles et isolation physique des ports.
  • VLAN 60 Backup : NAS isolé, flux entrants uniquement — protection anti-ransomware.

6. VPN WireGuard (Accès Admin Distant)

Paramètres

  • Protocole : UDP, Port 51820 sur WAN pfSense.
  • Plage VPN : 10.10.10.0/24 (réseau virtuel WireGuard).
  • Accès autorisé : VLAN 40 (AD), VLAN 50 (Management) et VLAN 60 (Backup).
  • Accès interdit : VLAN 10 (DMZ) et VLAN 30 (Clients).

Justification

Le VPN est réservé aux administrateurs IT en télétravail. Il permet de gérer pfSense, l'AD, Wazuh et le serveur backup sans exposer RDP/SSH directement sur Internet.

7. Sauvegarde & Restauration (VLAN 60)

Architecture Backup

  • VLAN dédié : 192.168.10.48/29 — totalement isolé des autres zones.
  • NAS/Serveur Backup : 192.168.10.50 — BorgBackup ou équivalent.
  • Sources sauvegardées : VLAN 20 (Serveurs : File Server + MariaDB) et VLAN 40 (AD).
  • Supervision : Logs unidirectionnels vers Wazuh SIEM (192.168.10.42).

Règles de Sécurité

  • Flux entrants uniquement : Le VLAN 60 ne peut initier aucune connexion vers les autres VLANs.
  • Internet bloqué : Aucun accès Internet depuis le backup — protection anti-ransomware.
  • DMZ/Clients bloqués : VLAN 10 et 30 ne peuvent jamais atteindre le VLAN 60.
  • Principe 3-2-1 : Base pour évoluer vers 3 copies, 2 supports, 1 hors-site.

VLAN 20 → VLAN 60 : ALLOW backup ports (TCP 9102) — unidirectionnel

VLAN 40 → VLAN 60 : ALLOW backup ports (TCP 9102) — unidirectionnel

VLAN 60 → VLAN 50 : ALLOW 1514/1515 (logs Wazuh uniquement)

VLAN 60 → Tous autres VLANs : DENY ALL

VLAN 10/30 → VLAN 60 : DENY ALL

VLAN 60 → Internet : DENY ALL (anti-ransomware)

8. Évolutivité & Haute Disponibilité (Redondance)

Pour éliminer les points de défaillance uniques (SPOF) identifiés, les évolutions suivantes sont préconisées pour le futur :

1. Redondance Firewall (CARP)

  • Cluster pfSense : Mise en place de deux nœuds pfSense en mode Haute Disponibilité.
  • Protocole CARP : Utilisation d'IP virtuelles (VIP) flottantes pour les passerelles de chaque VLAN (y compris VLAN 60).
  • Synchronisation : Réplication automatique des sessions et de la configuration entre les unités.

2. Agrégation de Liens (LACP)

  • Lien Trunk : Remplacer le lien unique par un EtherChannel (LACP).
  • Résilience L2 : Basculement instantané sur le lien secondaire en cas de rupture physique.
  • Performance : Augmentation de la bande passante globale pour le routage inter-VLAN.

💡 Cette approche garantit que la panne d'un équipement physique n'entraîne pas l'arrêt total des services.
